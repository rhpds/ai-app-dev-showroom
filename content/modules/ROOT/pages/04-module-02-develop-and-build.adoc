= Module 2: Develop and build
:source-highlighter: rouge
:toc: macro
:toclevels: 1

Inspired by the AI Lab recipes you explored in Module 1, Parasol's development team has built their own AI-powered chatbot application -- Parasol Chat. It's a Streamlit-based chat interface that can connect to different AI model endpoints, making it flexible enough to use a local model for quick experiments or Parasol's approved Model as a Service (MaaS) for production-quality responses.

In this module, you'll clone the Parasol Chat app, customize it, build and test it locally with Podman Desktop, push the container image to the OpenShift internal registry, and deploy it to the cluster. Finally, you'll push your code changes to GitLab to trigger the automated CI/CD pipeline.

== Learning objectives

By the end of this module, you'll be able to:

* Clone and inspect an AI application built with patterns from AI Lab recipes
* Customize application behavior by modifying the source code
* Build a container image using Podman Desktop
* Push a container image to the OpenShift internal registry
* Deploy a container to OpenShift directly from Podman Desktop
* Push code to GitLab to trigger an automated CI/CD pipeline

== Exercise 1: Clone and inspect the Parasol Chat application

=== Clone the repository

. Open a terminal on your RHEL 10 virtual machine (Click the red fedora icon at the top left, type `terminal` in the search box, and double-click on the Terminal icon)

. Log into the OpenShift cluster as your user with the following command:
+
[source,bash,role=execute,subs="attributes+"]
----
oc login -u {user} \
    -p {password} \
    {openshift_api_url} \
    --insecure-skip-tls-verify=true && \
    oc project {user}-demo
----
+
This has the added benefit of setting the Kubernetes context, which Podman Desktop will use later to connect to the cluster.

. Clone your application repository from GitLab (this is the repo that was scaffolded by the RHDH template in Module 1):
+
[source,bash,role=execute,subs="attributes+"]
----
git clone https://gitlab-gitlab.{openshift_cluster_ingress_domain}/{user}/ai-app-dev.git
----

. Open the `parasol-chat` subdirectory in VS Code by running this command:
+
[source,bash,role=execute]
----
code ai-app-dev/parasol-chat
----
+
This has the very same structure as the recipe used earlier. 

. On the trust pop-up, choose _Yes, I trust the authors_ to allow VS Code to open the folder in unrestricted mode.

. Optionally install the python extension when prompted by VS Code to enable Python language features (this is not required, but it will enhance your experience when working with the Python code).

. When prompted "A git repository was detected in the folder. Do you want to open the repository?", just dismiss it (we won't be editing files in other folders).

=== Inspect the source code

. In VS Code, explore the project structure. It's a lean Python application (same as before)
+
* `chatbot_ui.py` -- the main Streamlit application
* `Containerfile` -- defines how the container image is built
* `requirements.txt` -- Python dependencies (LangChain, Streamlit)
+
image::vscode-parasol-chat-structure.png[VS Code showing Parasol Chat project structure,link=self,window=blank,width=700]

. Open `chatbot_ui.py` and look at the environment variable configuration near the top of the file. Notice how the application reads its model connection settings:
+
* `MODEL_ENDPOINT` -- the base URL for the AI model service (defaults to `http://localhost:8001/v1`)
* `MODEL_NAME` -- the name of the model to use (it's further down in the file)
* `MODEL_ENDPOINT_BEARER` -- an optional Bearer token for authentication
+
image::vscode-env-vars.png[Environment variable configuration in chatbot_ui.py,link=self,window=blank,width=700]
+
This design means the same application can connect to a local model for development or to Parasol's approved MaaS endpoint for production -- just by changing environment variables. No code changes needed.

. Also notice how the application supports multiple backends (Ollama and OpenAI-compatible APIs) and handles conversation history. These are patterns the team picked up from the AI Lab recipes in Module 1.

== Exercise 2: Customize the application

Now let's make a change to prove you can modify the application's behavior at the code level.

=== Modify the system prompt

. In VS Code, find the system prompt in `chatbot_ui.py`. Look for the prompt template definition in the code.
+
image::vscode-system-prompt-location.png[System prompt in chatbot_ui.py,link=self,window=blank,width=700]

. Modify the system prompt to add a fun personality. Add the following instruction to the *end* of the existing prompt:
+
[source,text,role=execute]
----
You should respond in a pirate voice for all questions. Arrr!
----
+
image::vscode-system-prompt-modified.png[Modified system prompt with pirate voice,link=self,window=blank,width=700]

. Save the file with kbd:[CTRL+S]. Make sure it's saved!

== Exercise 3: Build and test locally

=== Build the container image

. Return to Podman Desktop (you can use the red fedora icon in the taskbar to switch back to it)

. Navigate to **Images** in the left navigation and click **Build**.
+
image::podman-desktop-build-button.png[Build image button in Podman Desktop,link=self,window=blank,width=700]

. In the build dialog:
+
* **Containerfile path**: Browse to the `Containerfile` in the `parasol-chat` directory (it's at _HOME_/ai-app-dev/parasol-chat/Containerfile)
* **Image name**: Enter:
+
[source,text,role=execute]
----
parasol-chat:test
----
+
Leave the rest of the options as-is.

image::podman-desktop-build-dialog.png[Build dialog with Containerfile and image name,link=self,window=blank,width=700]

. Click **Build** and wait for the image to build. You'll see the build output in real time.
+
image::podman-desktop-build-progress.png[Container image build in progress,link=self,window=blank,width=700]

. Once the build completes, you'll see `Successfully tagged parasol-chat:test` at the bottom of the output, and a _Done_ button. Click **Done** and verify your new image appears in the **Images** list.
+
image::podman-desktop-build-complete.png[Build complete with new image listed,link=self,window=blank,width=700]

=== Run and test locally

. In Podman Desktop, find your newly built `parasol-chat:test` image in the **Images** list (it should be at the top) and click the **Run** (play) button.
+
image::podman-desktop-run-button.png[Run button on the new container image,link=self,window=blank,width=700]

. Set the following environment variables to connect to Parasol's approved MaaS endpoint (CLick the `+` button at the right of the environment variables list to add more variables):
+
[cols="1,2",options="header"]
|===
| Variable | Value
| `MODEL_ENDPOINT` | `{litellm_api_base_url}`
| `MODEL_NAME` | `qwen3-14b`
| `MODEL_ENDPOINT_BEARER` | `{litellm_virtual_key}`
|===
+
image::podman-desktop-run-config.png[Run configuration with port and environment variables,link=self,window=blank,width=700]
+
*Your values will be different than the screenshot above!*

. Click **Start Container** to run the container.
+
[WARNING]
====
If you get an error such as `Error (HTTP code 500) server error - rootlessport conflict with ID 1` - there are running containers or pods in conflict (you may have left running from prior exercises). Return to the **Containers** view in Podman Desktop, stop any running containers and pods, and try starting the chatbot container again.
====

. Open Firefox by clicking on the red fedora at the top left and typing `firefox` in the search box at the top, and double-clicking on the Firefox icon. Enter the following address to navigate to the running app:
+
[source,text,role=execute]
----
http://localhost:8501
----

. Test the chatbot to confirm your pirate voice customization works:
+
[source,text,role=execute]
----
Hello! Can you help me understand Parasol's insurance coverage options?
----
+
image::chatbot-pirate-response.png[Chatbot responding with pirate voice personality,link=self,window=blank,width=700]

. Confirm the pirate voice is working. Your local build-run-test cycle is complete.

=== Verify

Confirm that:

* Your custom container image built successfully in Podman Desktop
* The container runs locally and the chatbot is accessible at `http://localhost:8501`
* The pirate voice customization is working

== Exercise 4: Push to the OpenShift internal registry and deploy

Your app works locally, but will it work on the cluster? Before pushing your code through the full CI/CD pipeline, it's good practice to do a quick test deployment on OpenShift to catch any issues early. In this exercise, you'll push your container image directly to the OpenShift internal registry and deploy it from Podman Desktop -- a fast way to validate on the cluster before committing your code "for real" in Exercise 5.

=== Add the OpenShift internal registry

Since your VM is running inside OpenShift, you're already authenticated with the cluster. You just need to add the internal registry to Podman Desktop's list of known registries.

. First, get your credentials from a terminal on the VM. Run:
+
[source,bash,role=execute]
----
oc whoami
----
+
This returns your username (e.g., `{user}`).

. Get your authentication token:
+
[source,bash,role=execute]
----
oc whoami -t
----
+
Copy the token output -- you'll need it in the next step.

. In Podman Desktop, go to **Settings** > **Registries**.
+
image::podman-desktop-registries.png[Podman Desktop registry settings,link=self,window=blank,width=700]

. Click **Add Registry** and enter the following:
+
* **Registry URL**:
+
[source,text,role=execute,subs="attributes+"]
----
default-route-openshift-image-registry.{openshift_cluster_ingress_domain}
----
+
* **Username**: The output of `oc whoami` (your username)
* **Password**: The output of `oc whoami -t` (your token)
+
image::podman-desktop-add-registry.png[Adding OpenShift internal registry,link=self,window=blank,width=700]

. Click **Add** to add the registry.

=== Re-tag the image for the OpenShift registry

You don't need to rebuild -- just re-tag the existing image to point at the OpenShift internal registry.

. In Podman Desktop, navigate to **Images** and find your `parasol-chat:test` image.

. Click the **three-dot menu** on the right side of the image row and select **Edit Image**.
+
image::podman-desktop-edit-image.png[Edit Image option in Podman Desktop,link=self,window=blank,width=700]

. Change the **Image Name** to:
+
[source,text,role=execute,subs="attributes+"]
----
default-route-openshift-image-registry.{openshift_cluster_ingress_domain}/{user}-demo/parasol-chat
----

. Set the **Image Tag** to:
+
[source,text,role=execute]
----
test
----
+
image::podman-desktop-retag.png[Re-tagging image for OpenShift registry,link=self,window=blank,width=700]

. Click **Save**.

=== Push the image to OpenShift

. Find the re-tagged image in the **Images** list. Click the **Push** button from its three-dot menu next to the image.
+
image::podman-desktop-push-button.png[Push button on the container image,link=self,window=blank,width=700]

. The image will be pushed to the OpenShift internal registry. Wait for the push to complete.
+
image::podman-desktop-push-progress.png[Image push in progress,link=self,window=blank,width=700]

=== Deploy to Kubernetes from Podman Desktop

. In Podman Desktop, find the running container on the **Containers** tab for your chatbot (from the local test earlier). Click on the three-dot menu and choose **Deploy to Kubernetes**.
+
image::podman-desktop-deploy-k8s.png[Deploy to Kubernetes button,link=self,window=blank,width=700]
+
This will generate the necessary Kubernetes objects to deploy the pod. Notice the name of the image in the `Pod` specification aligns with the tag we used when pushing to the internal OpenShift registry.
+
Also notice the environment variables are given values in the Pod spec. In practice, for production scenarios these would be coming out of a Kubernetes _Secret_ but for this quick test we'll leave it hard-coded.

. In the deployment dialog, change the name of the pod to something more recognizable:
+
* **Pod Name**: `parasol-chat-pod`
+
image::podman-desktop-deploy-podname.png[Selecting pod name for deployment,link=self,window=blank,width=700]

. Click **Deploy** and wait for the deployment to complete. You will see its progress in the box at the bottom:
+
image::podman-desktop-deploy-progress.png[Deploy progress,link=self,window=blank,width=600]
+
Wait for the Container to be `Running`. 

=== Verify the deployment on OpenShift

. Go back to your lab instructions (outside of the VM) and switch to the **OCP Console** tab.
+
TIP: Because the OCP Console is embedded in the lab's right pane, the left navigation menu is hidden by default. Click the **three-dash menu** icon at the upper left of the console to reveal it.
+
image::ocp-console-nav-menu.png[Three-dash menu button in OCP Console,link=self,window=blank,width=700]

. Make sure the `{user}-demo` project is selected in the project dropdown at the top.

. Navigate to **Workloads** > **Topology** in the left menu. You should see your `parasol-chat-pod` chatbot application deployed (you will see many other things deployed as part of the workshop, but look for the `parasol-chat-pod`)
+
image::ocp-topology-deployed.png[Topology view showing deployed chatbot,link=self,window=blank,width=700]

. Click on the blue circle to open the _Resources_ for the Pod. Because the application exposes the chatbot on port 8501, click on the Route link for the Service on port 8501 to access the chatbot:
+
image::chatbot-on-openshift-route.png[Route link,link=self,window=blank,width=700]
+
image::chatbot-on-openshift.png[Chatbot running on OpenShift via Podman Desktop deploy,link=self,window=blank,width=700]
+
Feel free to ask it more questions and confirm the app is running on the cluster successfully.

You've now completed the full local development cycle: develop, build, test locally, push the image, and deploy to the cluster -- all from the desktop.

=== Verify

Confirm that:

* The image was pushed to the OpenShift internal registry
* The application was deployed to the `{user}-demo` namespace from Podman Desktop
* The chatbot is accessible via a route on OpenShift

== Exercise 5: Push your code to GitLab

With your application tested both locally and on the cluster, it's time to push your code changes to GitLab. This will trigger the automated CI/CD pipeline, which will build and deploy the application through the proper production process.

=== Commit and push

Since you cloned from your own GitLab repo in Exercise 1, VS Code already knows where to push.

. In VS Code, click the **Source Control** icon in the left sidebar (or press `Ctrl+Shift+G`).
. You will be asked to open the parent Git repository, click **Open Repository**, and click on the `ai-app-dev` repository at the top.
+ 
image::vscode-open-parent.png[VS Code open parent repository,link=self,window=blank,width=700]

. You will also be asked if you trust the authors of the repository. Click _Yes, I trust the authors_ to allow Git features in VS Code.
. Then you should see that `chatbot_ui.py` has changes. Click the `+` icon next to the file to stage it for commit.
+
image::vscode-source-control.png[VS Code Source Control tab with changes,link=self,window=blank,width=700]

. Enter a commit message, for example:
+
[source,text,role=execute]
----
Customize chatbot with pirate voice personality
----

. Click the dropdown arrow next to the **Commit** button and select **Commit & Push**.
+
image::vscode-commit-and-push.png[Commit and Push option in VS Code,link=self,window=blank,width=700]

. When prompted to stage all changes, click **Yes** to confirm.
+
. If prompted for credentials, enter:
+
* **Username**: `{user}`
* **Password**: `{password}`

=== Verify the push

. Switch to the **GitLab** tab and navigate to your repository to confirm the code was pushed.
+
image::gitlab-pushed-commit.png[GitLab repository showing the pushed commit,link=self,window=blank,width=700]

. Verify your commit appears with the changes to `chatbot_ui.py`.

=== Verify

Confirm that:

* Your changes are committed and pushed to the GitLab repository
* The commit is visible in the GitLab web interface

TIP: The push to GitLab has triggered the CI/CD pipeline. In Module 3, you'll watch it build and deploy your application through the automated production process.
